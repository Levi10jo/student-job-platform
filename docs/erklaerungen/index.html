StudyWork – wie funktioniert das Ganze?

Diese Seiten erklären in einfachen Worten, wie unsere Web-App gebaut ist und warum wir es so gemacht haben. Auch ohne Programmier-Vorwissen verständlich.

Was ist StudyWork?

StudyWork ist eine Job-Plattform für Studenten. Unternehmen schreiben Stellen aus, Studenten suchen Jobs und bewerben sich. Technisch besteht die App aus drei Teilen, die zusammenarbeiten.

Die drei Teile – mit einer Restaurant-Analogie

Stell dir die App wie ein Restaurant vor. Dann sind die drei Teile:

🍽️ Frontend = der Gastraum

Alles, was der Gast sieht und anfasst: Speisekarte, Tische, Bestellzettel. Bei uns: die Web-Seiten im Browser (Buttons, Formulare, Job-Listen).

👨‍🍳 Backend = die Küche

Nimmt Bestellungen an, prüft sie, kocht und gibt sie zurück. Der Gast sieht die Küche nicht. Bei uns: der Server, der Anfragen bearbeitet und die Regeln durchsetzt.

🗄️ Datenbank = das Lager

Hier liegen alle Zutaten dauerhaft gelagert. Die Küche holt sie und legt Neues ab. Bei uns: MySQL speichert Firmen, Jobs, Bewerbungen usw.

Der Gast (Frontend) redet nie direkt mit dem Lager (Datenbank) – immer über die Küche (Backend). Genau so ist es bei uns: Das Frontend fragt das Backend, und nur das Backend spricht mit der Datenbank. Das sorgt für Ordnung und Sicherheit.

Der Weg einer Aktion (Beispiel: „Bewerbung absenden")

1. FRONTEND   Du füllst das Bewerbungsformular aus und klickst „Absenden".
              → Der Browser schickt die Daten an das Backend.

2. BACKEND    Der Server prüft: Sind alle Felder da? Ist die E-Mail gültig?
              Existiert die Stelle? Hast du dich schon beworben?
              → Wenn alles passt, gibt er die Daten weiter.

3. DATENBANK  MySQL speichert die Bewerbung dauerhaft in der Tabelle
              „applications".

4. BACKEND    Antwortet dem Browser: „Erfolg" (oder eine Fehlermeldung).

5. FRONTEND   Zeigt dir „Bewerbung gesendet!" an.

Details zu jedem Schritt stehen auf den drei Seiten: Frontend · Backend · Datenbank.

Die Technologien in einem Satz

TeilWomit gebautKurz erklärt
FrontendHTML, CSS, JavaScriptHTML = Struktur, CSS = Aussehen, JavaScript = Verhalten/Interaktion
BackendNode.js + ExpressEin selbst geschriebener Server, der die REST-API bereitstellt
DatenbankMySQLSpeichert alle Daten dauerhaft in Tabellen

Frontend und Backend laufen auf demselben Port (3000): Der Server liefert auch die Web-Seiten aus. Man ruft die App im Browser unter http://localhost:3000 auf.

Warum diese Aufteilung in drei Teile?

Weil jeder Teil eine klare Aufgabe hat. Das macht die App übersichtlich (jeder weiß, wo was passiert), sicher (nur das Backend darf an die Datenbank) und wartbar (man kann das Aussehen ändern, ohne die Datenbank anzufassen – und umgekehrt).

So liest du weiter: Oben kannst du jederzeit zwischen den drei Bereichen wechseln. Jede Seite erklärt wie wir den Teil gebaut haben und warum.
