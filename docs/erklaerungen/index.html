1. Was ist StudyWork?

StudyWork ist eine Job-Plattform für Studenten – eine Website, auf der zwei Gruppen etwas tun können:

  • Unternehmen registrieren sich, schreiben Stellenanzeigen aus und verwalten die eingehenden Bewerbungen (ansehen, Status ändern, Bewerberprofil öffnen).
  • Studenten suchen und filtern Jobs, bewerben sich (mit oder ohne Konto), pflegen ein Profil mit Lebenslauf und können Suchen als „Job-Alert" speichern.

„Web-App" heißt: Es ist keine Datei, die man installiert, sondern eine Anwendung, die im Browser läuft (Chrome, Safari, Firefox …). Man tippt eine Adresse ein und benutzt sie wie eine normale Website – nur dass sie auf Eingaben reagiert, Daten speichert und für jeden Nutzer etwas anderes anzeigt.

2. Die drei Teile – mit einer Restaurant-Analogie

Fast jede moderne Web-App ist in drei Schichten aufgebaut. Man versteht sie am schnellsten über ein Restaurant:

🍽️ Frontend = der Gastraum

Alles, was der Gast sieht und anfasst: Speisekarte, Tische, der Zettel, auf dem er bestellt. Bei uns: die Web-Seiten im Browser – Buttons, Formulare, Job-Listen, Farben.

→ Frontend im Detail

👨‍🍳 Backend = die Küche

Nimmt Bestellungen entgegen, prüft sie („haben wir das? ist die Bestellung vollständig?"), bereitet sie zu und reicht das Ergebnis hinaus. Der Gast betritt die Küche nie. Bei uns: der Server – ein Programm, das Anfragen bearbeitet und alle Regeln durchsetzt.

→ Backend im Detail

🗄️ Datenbank = das Lager

Hier liegt alles dauerhaft: Zutatenbestand, Lieferlisten. Nur die Küche darf ins Lager. Bei uns: MySQL – dort sind Unternehmen, Studenten, Jobs und Bewerbungen gespeichert, auch nach einem Neustart.

→ Datenbank im Detail

Die wichtigste Regel

Der Gast (Frontend) redet nie direkt mit dem Lager (Datenbank) – immer über die Küche (Backend). Genauso bei uns: Der Browser kann die Datenbank gar nicht erreichen. Er stellt Anfragen an das Backend, und nur das Backend liest und schreibt Daten. Das ist der zentrale Sicherheits- und Ordnungsgedanke der ganzen Architektur.

3. Ein paar Grundbegriffe (in 60 Sekunden)

BegriffBedeutung – einfach erklärt
ServerEin Programm, das dauerhaft läuft und auf Anfragen wartet – wie ein Kellner im Dienst. („Server" kann auch den Computer meinen, auf dem so ein Programm läuft.)
ClientDerjenige, der anfragt – bei uns der Browser.
HTTPDie „Sprache", in der Browser und Server miteinander reden. Jede Nachricht ist eine Anfrage (Request) oder eine Antwort (Response).
URL / AdresseSagt, wen und was man erreichen will, z. B. http://localhost:3000/jobs.html.
localhostBedeutet „dieser Computer hier". Die App läuft bei uns lokal – Server und Browser sind auf demselben Rechner.
PortEine Art „Zimmernummer" auf einem Computer. Viele Programme können gleichzeitig laufen; unser Server wartet auf Port 3000. Deshalb: localhost:3000.
JSONEin einfaches Textformat, in dem Frontend und Backend Daten austauschen. Sieht so aus: {"name": "Lena", "ort": "Berlin"} – Paare aus Bezeichnung und Wert.

4. Der komplette Weg einer Aktion

Unser Beispiel durch alle Erklärseiten: Lena bewirbt sich auf einen Job. Das passiert dabei – Schritt für Schritt:

  1. Lena füllt das Formular aus (Name, E-Mail, Anschreiben) und klickt „Bewerbung absenden". Das Frontend prüft sofort: Sind die Pflichtfelder ausgefüllt? Sieht die E-Mail gültig aus? Wenn nicht, erscheint direkt am Feld eine Fehlermeldung – ohne dass irgendetwas verschickt wird.
  2. Der Browser schickt eine HTTP-Anfrage an den Server: „POST /api/v1/applications" – übersetzt: „Lege eine neue Bewerbung an, hier sind die Daten (als JSON)."
  3. Das Backend prüft alles noch einmal – und strenger: Pflichtfelder? E-Mail-Format? Existiert der Job überhaupt? Ist die Stelle noch aktiv? Hat sich diese E-Mail auf diese Stelle schon einmal beworben? Erst wenn alles passt, geht es weiter. (Warum doppelt prüfen? Weil man dem Browser nie blind vertrauen darf – dazu mehr auf der Backend-Seite.)
  4. Das Backend speichert die Bewerbung in der Datenbank: In der Tabelle applications entsteht eine neue Zeile mit Lenas Daten, verknüpft mit dem Job, auf den sie sich beworben hat.
  5. Das Backend antwortet dem Browser mit einer Erfolgsmeldung (als JSON): {"success": true, "data": … } – oder, falls etwas nicht stimmte, mit einer verständlichen Fehlermeldung.
  6. Das Frontend zeigt das Ergebnis an: „Bewerbung gesendet!" – oder die Fehlermeldung als kleine Einblendung. Lena sieht die Bestätigung; die Daten liegen sicher im „Lager".

Merke: Frontend = zeigen & einsammeln, Backend = prüfen & entscheiden, Datenbank = dauerhaft aufbewahren.

5. Womit ist was gebaut?

TeilTechnologieKurz erklärtWo im Projekt
FrontendHTML, CSS, JavaScriptHTML = Struktur, CSS = Aussehen, JavaScript = Verhaltenfrontend/
BackendNode.js + Expressselbst geschriebener Server mit einer REST-Schnittstellebackend/
DatenbankMySQLspeichert alles dauerhaft in verknüpften Tabellenbackend/db_setup.sql, backend/db.js

Eine Besonderheit: Frontend und Backend laufen bei uns auf demselben Port. Der Server liefert nicht nur Daten, sondern auch die Web-Seiten selbst aus. Ruft man http://localhost:3000 auf, bekommt man die Startseite – ruft man http://localhost:3000/api/v1/jobs auf, bekommt man Daten.

Warum alles auf einem Port?

Wenn Seiten und Daten von verschiedenen Adressen kämen, würde der Browser das aus Sicherheitsgründen blockieren (Stichwort „CORS" – Browser misstrauen seitenübergreifenden Anfragen). Kommt beides vom selben Ort, gibt es dieses Problem gar nicht erst. Außerdem reicht so ein einziger Startbefehl für die ganze App.

6. Warum überhaupt drei Schichten?

  • Sicherheit: Alles, was der Nutzer direkt berühren kann (das Frontend), kann er auch manipulieren. Die verbindlichen Regeln sitzen deshalb im Backend – dahinter liegt die Datenbank, die von außen gar nicht erreichbar ist.
  • Übersicht: Jeder weiß sofort, wo er suchen muss: Aussehen? → Frontend. Regel oder Rechenlogik? → Backend. Gespeicherte Daten? → Datenbank.
  • Austauschbarkeit: Man könnte das Aussehen komplett erneuern, ohne die Datenbank anzufassen – oder später eine Handy-App bauen, die dasselbe Backend benutzt.
  • Teamarbeit: Zwei Leute können am Frontend arbeiten, zwei am Backend, zwei an der Datenbank – ohne sich ständig in die Quere zu kommen. Genau so ist unsere Gruppe aufgeteilt.

7. Wie starte ich die App selbst?

Die Schritt-für-Schritt-Anleitung steht in der README.md im Projektordner. Grob:

  1. Node.js und MySQL installieren (die zwei „Motoren" für Backend und Datenbank),
  2. die Datenbank mit backend/db_setup.sql anlegen (erzeugt Tabellen + Beispieldaten),
  3. im Ordner backend/ die Pakete installieren und den Server starten,
  4. im Browser http://localhost:3000 öffnen – fertig.

Demo-Zugänge (Passwort immer studywork123): z. B. Unternehmen kontakt@technova.de, Studentin lena.hofmann@uni-berlin.de.

🗣️ So erklärst du das Gesamtbild in 5 Sätzen weiter

  1. StudyWork ist eine Job-Plattform für Studenten, die als Web-App im Browser läuft.
  2. Sie besteht aus drei Teilen: Frontend (was man sieht), Backend (der Server, der alles prüft und entscheidet) und Datenbank (wo alles dauerhaft gespeichert ist).
  3. Der Browser redet nie direkt mit der Datenbank – jede Aktion geht als HTTP-Anfrage an das Backend, das prüft und dann speichert oder liest.
  4. Gebaut ist das mit HTML/CSS/JavaScript (Frontend), Node.js + Express (Backend) und MySQL (Datenbank); alles läuft lokal auf localhost:3000.
  5. Diese Trennung macht die App sicher, übersichtlich und gut im Team entwickelbar.
