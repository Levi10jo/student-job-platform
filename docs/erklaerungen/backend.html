Backend – der Server hinter den Kulissen

Wie unser selbst geschriebener Server Anfragen entgegennimmt, prüft, entscheidet und mit der Datenbank arbeitet – und warum wir es genau so gebaut haben.

🎯 Nach dieser Seite kannst du erklären:

  • was ein Server, Node.js und Express sind – und was beim Start unseres Servers passiert,
  • was eine REST-API ist: Adressen, die 5 Methoden, Statuscodes, JSON-Antworten,
  • wie eine Anfrage im Code wirklich bearbeitet wird (Zeile für Zeile am Bewerbungs-Beispiel),
  • wie Login mit Passwort-Hash, Sitzung und Cookie funktioniert,
  • wovor uns Prepared Statements, Rechteprüfungen und .env schützen – und was Swagger ist.

1. Was ist ein Server – und was sind Node.js und Express?

  • Ein Server ist ein Programm, das dauerhaft läuft und auf Anfragen wartet – wie ein Kellner im Dienst. Er „horcht" auf einer bestimmten Port-Nummer (bei uns 3000). Kommt eine Anfrage an localhost:3000, ist er zuständig.
  • Node.js ist die Laufzeitumgebung, mit der man solche Programme in JavaScript schreiben kann – derselben Sprache wie im Frontend. Ein großer Praxisvorteil: Das ganze Team denkt in einer Sprache.
  • Express ist eine schlanke Bibliothek für Node.js, die einem das Grundgerüst abnimmt: „Wenn jemand diese Adresse mit dieser Methode aufruft, führe diesen Code aus." Ein vollständiges Beispiel aus unserem Server:
app.get('/api/v1/health', (req, res) => {
  res.json({ success: true, data: { status: 'ok' } });
});

app.get(…) – „registriere eine Anlaufstelle für GET-Anfragen".

'/api/v1/health' – die Adresse (der „Pfad"), unter der sie erreichbar ist.

(req, res) => { … } – die Bearbeitungsfunktion: req (Request) enthält alles über die Anfrage, res (Response) ist das Werkzeug zum Antworten.

res.json(…) – schickt die Antwort als JSON zurück. Dieser „Health-Check" sagt schlicht: „Ich lebe."

Warum zählt das als „selbst erstellter Webserver"?

Express nimmt uns nur das Zerlegen der HTTP-Nachrichten ab. Jede einzelne Anlaufstelle – was geprüft wird, welche Datenbank-Abfrage läuft, was zurückkommt – haben wir von Hand geschrieben und kommentiert. Es gibt keinen Code-Generator und keine fertige „Blackbox": Man kann für jede Funktion der App die Zeilen zeigen, die sie umsetzen. Genau das verlangt die Aufgabenstellung („Selbsterstellter Webserver / Node.js").

2. Aufbau der Backend-Dateien

backend/ ├── server.js ← Startpunkt: baut den Server zusammen (siehe unten) ├── db.js ← die Verbindung zur MySQL-Datenbank ├── auth.js ← Passwort-Verschlüsselung + Login-Sitzungen + Rechte-Prüfer ├── helpers.js ← gemeinsame Hilfsfunktionen (einheitliche Antworten, Eingabe-Prüfungen) ├── db_setup.sql ← legt Datenbank, Tabellen und Beispieldaten an └── routes/ ← die eigentlichen Anlaufstellen, nach Thema sortiert ├── jobs.js ← Stellenanzeigen (7 Anlaufstellen) ├── applications.js ← Bewerbungen (5) ├── companies.js ← Unternehmen (5) ├── auth.js ← Registrieren / Login / Logout / „Wer bin ich?" (4) └── students.js ← Studenten-Profil, Lebenslauf, Job-Alerts (11)

Warum diese Aufteilung?

Ein Thema = eine Datei. Wer etwas an Bewerbungen ändern will, öffnet applications.js und ist fertig. Gemeinsames (z. B. „wie sieht eine Fehlermeldung aus?", „ist diese E-Mail gültig?") liegt genau einmal in helpers.js, statt fünfmal kopiert zu werden. Das macht den Code wartbar und verhindert, dass sich Kopien auseinanderentwickeln.

3. Was passiert beim Start? (server.js, Schritt für Schritt)

  1. Konfiguration laden: Aus der Datei .env werden Einstellungen gelesen (Port, Datenbank-Zugangsdaten). So stehen Passwörter nie im Quellcode.
  2. JSON-Übersetzer einschalten: express.json() wandelt ankommende JSON-Daten automatisch in JavaScript-Objekte um. Das ist eine sogenannte Middleware – ein Zwischenschritt, den jede Anfrage durchläuft, bevor sie bei ihrer Anlaufstelle ankommt (wie die Sicherheitskontrolle am Flughafen: alle müssen durch).
  3. Anlaufstellen einhängen: Die fünf Routen-Dateien werden unter ihren Adressen registriert – /api/v1/jobs, /api/v1/applications, /api/v1/companies, /api/v1/auth, /api/v1/students.
  4. Swagger aktivieren: Unter /api-docs wird die interaktive API-Dokumentation bereitgestellt (Abschnitt 8).
  5. Frontend ausliefern: express.static(frontend/) sorgt dafür, dass derselbe Server auch die HTML/CSS/JS-Dateien ausliefert. Deshalb läuft alles auf einem Port – und der Browser hat keine Sicherheitsbedenken wegen fremder Quellen.
  6. Auffangnetze spannen: Unbekannte /api/…-Adressen bekommen eine JSON-Fehlermeldung, unbekannte Seiten die freundliche 404.html; ein zentraler Fehler-Handler verhindert, dass z. B. kaputtes JSON den Server abstürzen lässt.
  7. Erst Datenbank prüfen, dann öffnen: Der Server testet die DB-Verbindung (ping). Nur wenn sie steht, öffnet er Port 3000 – sonst bricht er mit einer klaren Meldung ab. Warum? Ein Server ohne Datenbank würde bei jeder Anfrage scheitern; besser sofort und verständlich scheitern als später mysteriös.

4. Die REST-API – das Kernstück

Eine API (Programmier-Schnittstelle) ist die Liste der „Bestellnummern", die man beim Server aufrufen kann. REST ist die verbreitetste Ordnung dafür, mit zwei Grundideen:

  1. Dinge haben Adressen. „Alle Jobs" sind /api/v1/jobs, „der Job mit der Nummer 5" ist /api/v1/jobs/5.
  2. Die Methode sagt, was man will. Dieselbe Adresse kann Verschiedenes bedeuten – je nachdem, wie man anfragt:
MethodeBedeutungBeispiel bei StudyWork
GETlesenGET /api/v1/jobs → Jobliste holen
POSTneu anlegenPOST /api/v1/applications → Bewerbung einreichen
PUTkomplett ersetzenPUT /api/v1/jobs/5 → Job 5 vollständig überarbeiten
PATCHteilweise ändernPATCH /api/v1/applications/3 → nur den Status ändern
DELETElöschenDELETE /api/v1/jobs/5 → Anzeige entfernen

Unsere API nutzt alle fünf Methoden (Pflichtanforderung) über insgesamt 33 Anlaufstellen.

Anatomie einer Adresse

GET /api/v1/jobs?location=Berlin&job_type=Werkstudent
     └──┬──┘└─┬─┘ └──────────────┬────────────────┘
      Prefix  Ding        Filter („Query-Parameter")

GET /api/v1/jobs/5
                 └── Pfad-Parameter: die ID eines bestimmten Jobs
  • /api/v1/ – Vorsilbe aller Daten-Anlaufstellen. Das v1 ist die Versionsnummer: Käme später eine unverträgliche neue API, könnte sie als v2 parallel laufen, ohne Altes zu zerstören.
  • Query-Parameter (nach dem ?) sind optionale Filter.
  • Pfad-Parameter (im Code als :id notiert) wählen ein konkretes Exemplar aus.

Einheitliche Antworten + Statuscodes

Jede Antwort hat dieselbe Form – das Frontend muss nie raten:

Erfolg:  { "success": true,  "data": … }
Fehler:  { "success": false, "error": "Verständliche Meldung auf Deutsch" }

Zusätzlich trägt jede Antwort einen genormten Statuscode:

CodeBedeutungBeispiel
200OKJobliste erfolgreich geliefert
201Created – neu angelegtBewerbung gespeichert
400Bad Request – Eingabe fehlerhaftE-Mail ungültig, Pflichtfeld leer
401Unauthorized – nicht eingeloggtDashboard-Daten ohne Login angefragt
403Forbidden – eingeloggt, aber nicht berechtigtfremde Stellenanzeige löschen wollen
404Not FoundJob-Nummer existiert nicht
500Interner Serverfehlerunerwarteter Fehler – ohne interne Details nach außen

Merkhilfe für 401 vs. 403: 401 = „Wer bist du überhaupt?" (kein Ausweis), 403 = „Ich kenne dich – aber das darfst du nicht."

5. Eine Anfrage im Code – das Lena-Beispiel, Zeile für Zeile

So sieht (leicht gekürzt) die Anlaufstelle aus, die Lenas Bewerbung entgegennimmt:

router.post('/', async (req, res) => {
  try {
    const { job_id, student_name, student_email, cover_letter } = req.body;

    // 1) Eingaben prüfen
    if (!isPositiveInt(job_id))          return sendError(res, 400, 'job_id ungültig.');
    if (!isNonEmptyString(student_name)) return sendError(res, 400, 'Name fehlt.');
    if (!isValidEmail(student_email))    return sendError(res, 400, 'E-Mail ungültig.');

    // 2) Fachregeln prüfen (gegen die Datenbank)
    //    – existiert der Job und ist er „aktiv"?
    //    – gibt es schon eine Bewerbung mit dieser E-Mail auf diesen Job?

    // 3) Speichern – sicher, mit Platzhaltern (?)
    const [ergebnis] = await pool.execute(
      "INSERT INTO applications (job_id, student_name, student_email, cover_letter, status)
       VALUES (?, ?, ?, ?, 'offen')",
      [job_id, student_name.trim(), student_email.trim(), cover_letter || null]
    );

    // 4) Antworten
    return sendSuccess(res, 201, { id: ergebnis.insertId, … });
  } catch (fehler) {
    return sendServerError(res, fehler, 'POST /applications');   // → 500, Server läuft weiter
  }
});

router.post('/', …) – diese Funktion ist für POST /api/v1/applications zuständig. async erlaubt das Warten auf die Datenbank, ohne den Server zu blockieren – er kann währenddessen andere Anfragen bedienen.

req.body – die mitgeschickten JSON-Daten, bereits von der Middleware in ein Objekt übersetzt. Wir entnehmen die vier Felder.

Schritt 1 – Formprüfung: Hilfsfunktionen aus helpers.js prüfen Typ und Format. Bei Problemen antwortet sendError sofort mit Code 400 und einer verständlichen Meldung – der Rest der Funktion läuft dann gar nicht mehr.

Schritt 2 – Fachregeln: Dinge, die nur die Datenbank weiß: Gibt es den Job? Nimmt er noch Bewerbungen an (Status „aktiv")? Doppelbewerbung mit derselben E-Mail? So etwas kann das Frontend nicht zuverlässig prüfen – deshalb gehört es hierher.

Schritt 3 – Speichern: pool.execute schickt den SQL-Befehl an MySQL. Die ? sind Platzhalter: Die Werte werden getrennt vom Befehl übergeben („Prepared Statement") – der zentrale Schutz vor SQL-Injection (Details auf der Datenbank-Seite). Der Status wird fest auf 'offen' gesetzt – ein Bewerber kann sich also nicht selbst als „angenommen" eintragen.

Schritt 4 – Antwort: Code 201 („neu angelegt") plus die neue Bewerbung inklusive ihrer vergebenen Nummer.

try/catch – passiert irgendetwas Unerwartetes (z. B. Datenbank kurz weg), fängt catch den Fehler: Antwort 500 mit neutraler Meldung, Details nur ins interne Log. Der Server stürzt nie wegen einer einzelnen Anfrage ab. Dieses Muster – prüfen → Regeln → Datenbank → antworten, alles in try/catch – hat jede unserer 33 Anlaufstellen.

6. Login & Rechte – selbst gebaut, Schritt für Schritt

Die Vorgabe erlaubt keinen externen Login-Dienst. Unser eigenes System (in backend/auth.js) besteht aus drei Bausteinen:

a) Passwörter: niemals im Klartext

  • Beim Registrieren wird das Passwort durch eine Hash-Funktion (scrypt) geschickt – eine mathematische Einbahnstraße: Aus dem Passwort wird ein langer Zeichensalat, aus dem sich das Passwort nicht zurückrechnen lässt. Nur dieser Hash wird gespeichert.
  • Beim Login wird die Eingabe genauso „zerhackt" und mit dem gespeicherten Hash verglichen. Stimmen beide überein, war das Passwort richtig – ohne dass es je gespeichert wurde.
  • Zusätzlich bekommt jedes Passwort ein zufälliges „Salz" (Salt) beigemischt: Dadurch ergeben selbst zwei identische Passwörter völlig verschiedene Hashes – vorgefertigte „Nachschlagetabellen" von Angreifern laufen ins Leere. Und scrypt ist mit Absicht langsam, damit massenhaftes Durchprobieren unbezahlbar teuer wird.

b) Sitzungen: eingeloggt bleiben trotz „vergesslichem" HTTP

HTTP hat kein Gedächtnis – jede Anfrage ist für den Server erst einmal ein Fremder. So lösen wir das:

  1. Nach erfolgreichem Login erzeugt der Server eine lange Zufallsnummer (das Sitzungs-„Token") und merkt sich intern: „Diese Nummer gehört zu Lena (Rolle: Student)."
  2. Die Nummer wird dem Browser als Cookie mitgegeben – ein kleiner Merkzettel, den der Browser bei jeder weiteren Anfrage an diesen Server automatisch mitschickt.
  3. Der Server schaut bei jeder Anfrage nach: „Kenne ich dieses Token?" – und weiß damit, wer anfragt und welche Rolle die Person hat.
  4. Beim Logout wird der Eintrag serverseitig gelöscht und das Cookie geleert – das Token ist ab sofort wertlos.

Zwei Schutzdetails: Das Cookie ist HttpOnly (JavaScript im Browser kann es nicht lesen – selbst eingeschleuster Code könnte es nicht stehlen) und SameSite (fremde Websites können es nicht für untergeschobene Anfragen missbrauchen).

Bewusste Vereinfachung: Unsere Sitzungen liegen im Arbeitsspeicher des Servers. Nach einem Server-Neustart muss man sich neu anmelden. Für ein Produktivsystem würde man sie in der Datenbank ablegen – für dieses Projekt ist die einfache, transparente Lösung die richtige (und sie ist dokumentiert).

c) Rollen und Eigentum: Wer darf was?

  • Vor geschützten Anlaufstellen sitzen zwei „Türsteher"-Funktionen (Middleware): requireCompany und requireStudent. Sie prüfen die Sitzung und lassen nur die richtige Rolle durch – sonst Antwort 401.
  • Danach folgt die Eigentums-Prüfung („Ownership"): Ein Unternehmen darf nur eigene Anzeigen bearbeiten und nur Bewerbungen auf eigene Stellen sehen – sonst 403. Bewerbungen enthalten persönliche Daten (Name, E-Mail); ohne diese Prüfung könnte Firma A die Bewerber von Firma B auslesen.
  • Öffentlich bleiben nur die Dinge, die öffentlich sein sollen: Jobs ansehen, Unternehmen ansehen, sich bewerben (auch ohne Konto), eine Anzeige melden.

8. Swagger – die interaktive API-Dokumentation

Läuft der Server, zeigt http://localhost:3000/api-docs eine automatisch erzeugte, klickbare Übersicht aller 33 Anlaufstellen – nach Themen gruppiert, mit Parametern, Beispielwerten und möglichen Antworten. Über „Try it out" kann man jede Anfrage direkt im Browser abschicken und die echte Antwort sehen.

Wie es funktioniert: Über jeder Anlaufstelle im Code steht ein kurzer, formalisierter Kommentar (@swagger …). Zwei Zusatzpakete (swagger-jsdoc + swagger-ui-express) sammeln diese Kommentare beim Start ein und bauen daraus die Oberfläche.

Warum Swagger?

Man kann die API verstehen und testen, ohne Code zu lesen – ideal für Teammitglieder, für die Präsentation und zum Debuggen. Weil die Doku im Code neben der Route steht, bleibt sie eher aktuell, als wenn sie in einem getrennten Dokument leben würde. (Sie war zudem ein optionales Bonus-Kriterium der Aufgabe.)

10. Typische Fragen – kurze Antworten

Warum Express und nicht „rohes" Node.js?
Express spart nur den Boilerplate-Teil (HTTP zerlegen, Pfade zuordnen). Die gesamte Logik ist handgeschrieben – der Lerneffekt bleibt, die Lesbarkeit steigt.
Was passiert bei einer Anfrage an eine Adresse, die es nicht gibt?
API-Pfade → JSON-Fehler mit 404; alles andere → unsere freundliche 404-Seite. Nichts stürzt ab.
Warum prüft der Server Dinge, die das Frontend schon geprüft hat?
Weil man Anfragen auch am Frontend vorbei schicken kann (z. B. mit curl oder Swagger). Verbindlich ist nur, was der Server prüft.
Warum wird nach einem Server-Neustart jeder ausgeloggt?
Die Sitzungen liegen bewusst im Arbeitsspeicher (einfach + transparent). Ein Neustart leert ihn. Produktiv würde man Sessions in der Datenbank halten.
Kann ein Bewerber seinen Bewerbungs-Status selbst auf „angenommen" setzen?
Nein. Beim Anlegen erzwingt der Server Status „offen"; ändern kann ihn nur das eingeloggte Unternehmen, dem die Stelle gehört (PATCH + Rechteprüfung).
Woher weiß der Server, dass die Anfrage von einem eingeloggten Unternehmen kommt?
Am mitgeschickten Sitzungs-Cookie: Token nachschlagen → Nutzer + Rolle bekannt → Türsteher-Middleware entscheidet.

🗣️ So erklärst du das Backend in 5 Sätzen weiter

  1. Das Backend ist unser selbst geschriebener Server (Node.js + Express), der auf Port 3000 läuft, alle Anfragen entgegennimmt und zusätzlich die Web-Seiten selbst ausliefert.
  2. Er stellt eine REST-API bereit: Dinge haben Adressen (/api/v1/jobs …), und die HTTP-Methode (GET/POST/PUT/PATCH/DELETE) sagt, ob gelesen, angelegt, geändert oder gelöscht wird – insgesamt 33 Anlaufstellen, Antworten immer als einheitliches JSON mit Statuscode.
  3. Jede Anlaufstelle arbeitet nach demselben Muster: Eingaben prüfen → Fachregeln prüfen → Datenbank-Zugriff per Prepared Statement → antworten; alles in try/catch, damit nichts den Server abstürzen lässt.
  4. Login ist selbst gebaut: Passwörter nur als scrypt-Hash mit Salt, Sitzungen über ein HttpOnly-Cookie, Türsteher-Middleware für die Rollen und Eigentums-Prüfungen, damit jeder nur an seine eigenen Daten kommt.
  5. Unter /api-docs dokumentiert Swagger alle Anlaufstellen interaktiv – erzeugt aus Kommentaren direkt im Code.
