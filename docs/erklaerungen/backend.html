Backend – die „Küche" hinter den Kulissen

Der selbst geschriebene Server: wie er Anfragen bearbeitet, die Regeln durchsetzt und mit der Datenbank redet – und warum wir es so gemacht haben.

In einem Satz

Das Backend ist ein selbst geschriebener Server mit Node.js und Express. Er stellt eine REST-API bereit (eine Sammlung von „Anlaufstellen", die das Frontend aufrufen kann), prüft alle Eingaben und spricht als Einziger mit der Datenbank.

Was ist ein „Server" – und was ist Node.js / Express?

  • Ein Server ist ein Programm, das ständig läuft und auf Anfragen wartet – wie ein Kellner, der auf Bestellungen wartet.
  • Node.js ermöglicht es, solche Server-Programme in der Sprache JavaScript zu schreiben.
  • Express ist eine schlanke Hilfs-Bibliothek, die das Schreiben eines Servers stark vereinfacht (z. B. „wenn jemand diese Adresse aufruft, tue Folgendes…").

Warum Node.js + Express?

So können wir Frontend und Backend in derselben Sprache (JavaScript) schreiben – das spart Umgewöhnung. Express ist bewusst „durchsichtig": jeder Bearbeitungs- Schritt ist von Hand geschrieben und nachvollziehbar (keine Blackbox). Genau das verlangt auch die Aufgabenstellung („selbst erstellter Webserver / Node.js").

Aufbau der Backend-Dateien

backend/ ├── server.js ← Startpunkt: startet den Server, verteilt Anfragen ├── db.js ← die Verbindung zur MySQL-Datenbank ├── auth.js ← Passwort-Verschlüsselung + Login-Sitzungen ├── helpers.js ← gemeinsame Hilfsfunktionen (Antworten, Prüfungen) ├── db_setup.sql ← legt die Datenbank + Beispieldaten an └── routes/ ← je Themenbereich eine Datei mit den „Anlaufstellen" ├── jobs.js ← Stellenanzeigen ├── applications.js ← Bewerbungen ├── companies.js ← Unternehmen ├── auth.js ← Registrieren / Login / Logout └── students.js ← Studenten-Profil, Lebenslauf, Job-Alerts

Warum in so viele Dateien aufteilen?

Damit jede Datei nur ein Thema behandelt. Sucht man etwas zu Bewerbungen, schaut man in applications.js – nicht in einer riesigen Datei. Das nennt man „Modularität" und macht den Code wartbar.

Was ist eine REST-API? (der Kern der Aufgabe)

Eine API ist eine Liste von „Anlaufstellen" (URLs), die man aufrufen kann, um etwas zu tun. REST ist eine verbreitete Art, diese Anlaufstellen zu ordnen: Man spricht Dinge (z. B. „Jobs") über eine Adresse an, und die Art der Anfrage (die HTTP-Methode) sagt, was man will. Wir nutzen alle fünf Methoden:

MethodeBedeutungBeispiel bei uns
GETDaten lesenAlle Jobs anzeigen
POSTNeu anlegenEine Bewerbung absenden
PUTKomplett ersetzenEinen Job vollständig überarbeiten
PATCHTeilweise ändernNur den Status einer Bewerbung ändern
DELETELöschenEine Anzeige entfernen

Alle unsere Anlaufstellen beginnen mit /api/v1/…, z. B. GET /api/v1/jobs („gib mir alle Jobs") oder POST /api/v1/applications („nimm diese neue Bewerbung an").

Analogie: Die REST-API ist wie eine Speisekarte mit Bestellnummern. Das Frontend „bestellt" per Nummer (URL + Methode), die Küche (Backend) weiß genau, was zu tun ist, und liefert das Ergebnis zurück.

Immer dieselbe Antwort-Form

Jede Antwort ist ein einheitliches Datenpaket (im Format „JSON"):

Erfolg:   { "success": true,  "data": ... }
Fehler:   { "success": false, "error": "Verständliche Meldung" }

So weiß das Frontend immer sofort, ob es geklappt hat. Dazu passende „Statuscodes": 200/201 = OK, 400 = falsche Eingabe, 401 = nicht eingeloggt, 403 = keine Berechtigung, 404 = nicht gefunden.

Was passiert bei einer Anfrage? (Ablauf)

Jede Anlaufstelle arbeitet nach demselben Muster:

  1. Eingaben prüfen: Sind alle Pflichtfelder da? Ist die E-Mail gültig? Ist der Text nicht zu lang? Wenn nicht → freundliche Fehlermeldung (Code 400).
  2. Rechte prüfen (falls nötig): Ist die Person eingeloggt? Gehört ihr das, was sie ändern will?
  3. Mit der Datenbank arbeiten: lesen oder speichern.
  4. Antworten: Erfolg mit Daten – oder eine klare Fehlermeldung.

Warum so streng prüfen?

Weil man Eingaben aus dem Internet nie blind vertrauen darf. Die Aufgabenstellung verlangt ausdrücklich, dass „Fehleingaben verhindert" werden. Deshalb prüfen wir doppelt: einmal im Frontend (für schnelles Feedback) und noch einmal im Backend (das ist die verbindliche Kontrolle).

Login & Rechte (einfach erklärt)

  • Passwörter werden nie im Klartext gespeichert, sondern „verschlüsselt" (genauer: als sogenannter Hash). Selbst wenn jemand die Datenbank stiehlt, sieht er keine echten Passwörter.
  • Login-Sitzung: Nach dem Anmelden bekommt der Browser einen geheimen „Stempel" (ein Cookie). Bei jeder weiteren Anfrage schickt er ihn mit, und der Server erkennt daran, wer man ist. Der Stempel ist so geschützt, dass ihn fremdes JavaScript nicht auslesen kann.
  • Zwei Rollen: „Unternehmen" und „Student" dürfen jeweils nur bestimmte Dinge – und immer nur ihre eigenen Daten. (Ein Unternehmen kann z. B. nur seine eigenen Anzeigen bearbeiten.)

Warum ein eigenes Login und kein fertiger Dienst?

Die Vorgabe erlaubt keine externen Anbieter. Deshalb haben wir Login und Sitzungen selbst gebaut – nur mit Node.js-Bordmitteln. Das ist zwar mehr Arbeit, aber vollständig nachvollziehbar und ohne fremde Abhängigkeit.

Swagger: die interaktive API-Doku

Wenn der Server läuft, gibt es unter http://localhost:3000/api-docs eine automatisch erzeugte, anklickbare Übersicht aller Anlaufstellen. Man kann dort jede Anfrage direkt ausprobieren („Try it out").

Warum Swagger?

Damit man die API sehen und testen kann, ohne den Code zu lesen – praktisch für die Präsentation und für Teammitglieder. Sie entsteht aus kurzen Kommentaren, die direkt über jeder Anlaufstelle im Code stehen, und ist bei uns vollständig (alle Methoden dokumentiert). Das war zusätzlich ein optionales Bonus-Kriterium der Aufgabe.
