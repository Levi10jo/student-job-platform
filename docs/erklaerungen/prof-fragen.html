Mögliche Prof-Fragen – mit Antworten

Fragen, die in der Präsentation realistisch kommen können – jeweils mit einer kurzen, korrekten Antwort und dem Ort im Code, auf den man zeigen kann.

Tipp fürs Antworten: Kurz antworten (2–4 Sätze), dann anbieten, es im Code oder live in der App zu zeigen. Die Datei-Angaben in den Antworten sind genau dafür da.

A. Architektur & Zusammenspiel

„Woher kommen die Job-Angebote auf der Seite? Werden die aus der Datenbank geholt?"
Ja. Beim Öffnen der Jobsuche schickt das JavaScript der Seite eine Anfrage GET /api/v1/jobs?status=aktiv an unseren Server. Der Server führt daraufhin eine SQL-Abfrage auf die MySQL-Tabelle jobs aus (mit JOIN auf companies, um den Firmennamen mitzuliefern) und schickt das Ergebnis als JSON zurück. Das Frontend baut daraus die Job-Kacheln. Zeigen: Frontend frontend/js/jobs.js (Aufruf API.jobs.list), Backend backend/routes/jobs.js (die SELECT-Abfrage).
„Wie kommunizieren Frontend und Backend genau?"
Über HTTP-Anfragen mit JSON-Daten – das Frontend nutzt die Browser-Funktion fetch. Alle Aufrufe laufen zentral über frontend/js/api.js; die Antworten haben immer dasselbe Format {"success": true/false, "data"/"error": …}. Der Browser redet nie direkt mit der Datenbank – nur der Server tut das.
„Was passiert Schritt für Schritt, wenn ich auf ‚Bewerbung absenden' klicke?"
1) Frontend prüft die Felder (Pflicht, E-Mail-Format). 2) POST /api/v1/applications mit den Daten als JSON. 3) Backend prüft erneut + Fachregeln (Job existiert? aktiv? schon beworben?). 4) INSERT in die Tabelle applications per Prepared Statement. 5) Antwort 201 mit dem neuen Datensatz. 6) Frontend zeigt die Bestätigung. Zeigen: frontend/js/jobs.js (Formular-Handler) und backend/routes/applications.js (POST-Route).
„Warum laufen Frontend und API auf demselben Port?"
Express liefert den frontend/-Ordner per express.static() mit aus (backend/server.js). Ein Ursprung für Seiten und Daten heißt: keine CORS-Probleme im Browser und nur ein Startbefehl für die ganze App.
„Kann man die API auch ohne das Frontend benutzen?"
Ja – z. B. mit curl oder direkt über die Swagger-Oberfläche unter http://localhost:3000/api-docs („Try it out"). Genau das zeigt, dass Frontend und Backend sauber getrennt sind.

B. Frontend

„Welches Framework habt ihr im Frontend benutzt?"
Keins – bewusst reines HTML, CSS und Vanilla-JavaScript, wie in der Aufgabenstellung vorgesehen. Bei dieser Projektgröße bleibt so jeder Schritt nachvollziehbar; gemeinsame Logik (API-Aufrufe, Navigation, Meldungen) haben wir selbst in frontend/js/api.js zentralisiert.
„Wie funktioniert die Live-Suche?"
Auf das Suchfeld ist ein input-Event-Listener registriert. Mit „Debounce" (ca. 300 ms Tipp-Pause) wird dann neu bei der API angefragt und die Liste neu gerendert. Die Filter werden zusätzlich in die URL geschrieben, damit Such-Links teilbar sind. Zeigen: frontend/js/jobs.js.
„Wie stellt ihr sicher, dass eingegebene Texte keinen Schaden anrichten (XSS)?"
Jeder Text, der von Nutzern/aus der DB kommt, läuft vor dem Einfügen ins HTML durch escapeHtml() (in api.js): Zeichen wie < werden in harmlose Entities umgewandelt – der Browser zeigt sie an, statt sie als Code auszuführen.
„Wie habt ihr Responsive Design umgesetzt?"
Mit CSS Media Queries in frontend/css/style.css: Unter bestimmten Breiten (≈880 px, ≈560 px) ändern sich Raster und Navigation. Getestet für Handy (~375 px), Tablet (768 px) und Desktop (1280 px+).
„Wie funktioniert der Dark-Mode?"
Der Umschalter setzt data-theme="dark" am <html>-Element; das CSS tauscht dann nur die Farb-Variablen. Die Wahl liegt im localStorage und wird beim Laden vor dem Rendern gesetzt (kein Aufblitzen).
„Woher weiß die Seite, ob jemand eingeloggt ist?"
Beim Laden fragt jede Seite GET /api/v1/auth/me. Der Server erkennt den Nutzer am Session-Cookie und antwortet mit Name/Rolle oder null; danach passt das Frontend Navigation und Inhalte an. Das Cookie selbst kann das JavaScript nicht lesen (HttpOnly).

C. Backend & REST

„Zeigen Sie mir, wo ein Request verarbeitet wird."
Startpunkt backend/server.js: Middleware (express.json) → Router-Mounts (/api/v1/jobs → routes/jobs.js usw.). In der Routen-Datei steht pro Endpunkt eine Handler-Funktion: validieren → ggf. Rechte prüfen → SQL per pool.execute → einheitliche JSON-Antwort. Alles in try/catch.
„Nutzt ihr wirklich alle fünf HTTP-Methoden? Wofür?"
Ja: GET (Jobs/Bewerbungen lesen), POST (Bewerbung/Job/Konto anlegen), PUT (Job oder Profil vollständig aktualisieren), PATCH (nur Status ändern, z. B. Bewerbung „gesehen"), DELETE (Job, Bewerbung, Konto löschen). Insgesamt 33 Endpunkte – alle in Swagger dokumentiert.
„Was ist der Unterschied zwischen PUT und PATCH – und wo sieht man das bei euch?"
PUT ersetzt die Ressource komplett (alle Felder Pflicht) – z. B. „Job bearbeiten" im Dashboard. PATCH ändert nur einzelne Felder – z. B. der Schnell-Statuswechsel einer Anzeige oder der Bewerbungsstatus. Zeigen: backend/routes/jobs.js (beide nebeneinander).
„Was bedeutet das /api/v1 in den Adressen?"
/api trennt Daten-Endpunkte von den Webseiten; v1 ist die API-Version. Käme später eine inkompatible Version, könnte sie als v2 parallel laufen, ohne alte Clients zu brechen.
„Welche Statuscodes gebt ihr zurück und wann?"
200 OK (gelesen/geändert), 201 Created (neu angelegt), 400 (Eingabe ungültig), 401 (nicht eingeloggt), 403 (eingeloggt, aber nicht berechtigt – z. B. fremder Job), 404 (nicht gefunden), 500 (unerwarteter Fehler, ohne interne Details). Merkhilfe: 401 = „Wer bist du?", 403 = „Das darfst du nicht."
„Wie verhindert ihr, dass sich jemand zweimal auf dieselbe Stelle bewirbt?"
Serverseitig: Vor dem INSERT prüft die POST-Route per SELECT, ob es zu diesem Job schon eine Bewerbung mit derselben E-Mail gibt → dann 400 mit klarer Meldung. Zusätzlich zeigt das Frontend eingeloggten Studenten „Bereits beworben" statt des Formulars – aber verbindlich ist die Server-Prüfung.
„Kann ein Bewerber seinen Status selbst auf ‚angenommen' setzen?"
Nein. Beim Anlegen erzwingt der Server Status offen (fest im INSERT). Ändern kann ihn nur via PATCH ein eingeloggtes Unternehmen – und nur für Bewerbungen auf die eigenen Stellen (Ownership-Prüfung).
„Was passiert, wenn die Datenbank beim Start nicht läuft?"
Der Server testet die Verbindung vor dem Portöffnen (testConnection() in db.js) und bricht mit einer klaren deutschen Fehlermeldung ab, statt später bei jeder Anfrage kryptisch zu scheitern.

D. Login & Sicherheit

„Wo und wie speichert ihr Passwörter?"
Nie im Klartext. In der DB steht nur ein scrypt-Hash mit zufälligem Salt (Spalte password_hash, Format scrypt:salt:hash). Beim Login wird die Eingabe gleich gehasht und zeitkonstant verglichen (timingSafeEqual). Zeigen: backend/auth.js.
„Wie funktioniert die Session? Woher weiß der Server, wer ich bin?"
Nach dem Login erzeugt der Server ein 32-Byte-Zufalls-Token, merkt es sich serverseitig (Token → Nutzer/Rolle) und gibt es dem Browser als HttpOnly-Cookie (sw_session). Der Browser schickt es automatisch mit; der Server schlägt es nach. HttpOnly heißt: JavaScript kann das Cookie nicht lesen → kein Token-Diebstahl per XSS.
„Warum ist man nach einem Server-Neustart ausgeloggt?"
Die Sessions liegen bewusst im Arbeitsspeicher (einfach + transparent, dokumentierte MVP-Vereinfachung). Produktiv würde man sie in der DB oder z. B. Redis halten – das wäre an genau einer Stelle (auth.js) austauschbar.
„Was ist SQL-Injection und wie schützt ihr euch?"
Ein Angriff, bei dem jemand SQL-Befehle in Eingabefelder schreibt (z. B. '; DELETE FROM jobs; --). Wir kleben Eingaben nie in SQL-Strings, sondern nutzen ausschließlich Prepared Statements: pool.execute('… WHERE id = ?', [wert]). Die DB behandelt Eingaben dadurch strikt als Daten, nie als Befehl.
„Wer darf Bewerberdaten sehen?"
Nur das Unternehmen, dem die zugehörige Stelle gehört – geprüft per Login-Middleware (requireCompany) plus Ownership-Check in jeder Route. Ohne Login: 401; fremde Daten: 403. Bewerber-Profile unterliegen zusätzlich der vom Studenten gewählten Sichtbarkeit.
„Wo liegen eure Zugangsdaten/Secrets?"
Nur in der lokalen .env-Datei, die per .gitignore vom Repository ausgeschlossen ist. Im Repo liegt nur die Vorlage .env.example mit Platzhaltern.

E. Datenbank

„Wie sieht euer Datenmodell aus?"
6 Tabellen: companies, students, jobs, applications, job_reports, job_alerts. Kernbeziehungen (je 1 : 0..*): Unternehmen→Jobs, Job→Bewerbungen, Job→Meldungen, Student→Alerts. Bewerbungen sind nur „weich" (über die E-Mail) mit Studenten verknüpft, weil Bewerben ohne Konto möglich ist. Details/UML: Seite Datenbank.
„Warum hat die Bewerbung keinen Fremdschlüssel auf den Studenten?"
Designentscheidung: Gast-Bewerbungen sollen möglich sein (niedrige Hürde). Name/E-Mail werden als Text gespeichert; hat der Bewerber ein Konto, ordnet das Backend es über die identische E-Mail zu (z. B. für „Meine Bewerbungen" und den Profil-Link im Dashboard).
„Was passiert in der DB, wenn ein Unternehmen sein Konto löscht?"
ON DELETE CASCADE-Kettenreaktion: seine Jobs werden gelöscht → dadurch automatisch deren Bewerbungen und Meldungen. Keine verwaisten Daten. (Beim Studenten löscht das Backend dessen Bewerbungen zusätzlich per E-Mail-Abgleich, da dort bewusst kein FK existiert.)
„Wie ist die Datenbank ans Backend angebunden?"
Über das Paket mysql2 mit einem Connection Pool (bis 10 wiederverwendete Verbindungen) in backend/db.js; Zugangsdaten aus .env; alle Abfragen als Prepared Statements über pool.execute().
„Woher kommen die Beispieldaten? Wie setze ich die DB auf?"
Aus backend/db_setup.sql – ein idempotentes Skript (löscht + erstellt Tabellen und füllt Seed-Daten: 6 Unternehmen, 20 Jobs, 1 Demo-Studentin, 5 Bewerbungen, 1 Alert). Einmal per mysql -u root < db_setup.sql ausführen; erneutes Ausführen setzt alles zurück.
„Kann in der Status-Spalte beliebiger Text stehen?"
Nein – ENUM-Spalten (Job-Status, Bewerbungs-Status, Job-Typ) erlauben nur die definierten Werte; die DB selbst lehnt alles andere ab. Zusätzlich validiert das Backend vorher.
„Wie bleibt die Suche bei vielen Jobs schnell?"
Indizes auf den Filter-Spalten (company_id, job_type, status, job_id, student_id) – wie ein Stichwortverzeichnis springt die DB direkt zu den passenden Zeilen.
„Wo liegt der hochgeladene Lebenslauf?"
Als base64-kodierte PDF in der Spalte students.cv_data (max. 3 MB, Magic-Bytes-Prüfung auf echtes PDF). Ausgeliefert wird er nur über eine zugriffsgeschützte Route (GET /api/v1/students/:id/cv) – dieselben Sichtbarkeitsregeln wie beim Profil. Für die Demo bewusst ohne extra Dateiserver.

F. Meta-Fragen zum Projekt

„Wie habt ihr getestet?"
Systematisch über die API (mit curl/Swagger: alle Methoden, Fehlerfälle wie 400/401/403/404, Schutzregeln wie Doppelbewerbung und Fremdzugriff) und im Browser entlang der Nutzer-Abläufe (Student-Weg, Unternehmens-Weg, responsive, Dark-Mode) – jeweils mit Kontrolle der Browser-Konsole.
„Was würdet ihr als Nächstes verbessern?"
Ehrlich benennen: Sessions in die DB (Neustart-fest), echte E-Mail-Zustellung für Job-Alerts (aktuell simuliert), Passwort-zurücksetzen, Moderations-Ansicht für gemeldete Anzeigen, Pagination bei sehr vielen Jobs, automatisierte Tests (z. B. Jest/Supertest).
„Ist die Seite öffentlich erreichbar?"
Nein, bewusst lokal (localhost:3000) – ein Deployment war nicht Teil der Aufgabe. Jeder kann sie aber per README in wenigen Minuten lokal starten; die Architektur wäre unverändert deploybar.
