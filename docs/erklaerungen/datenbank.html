1. Was ist eine (relationale) Datenbank – und warum brauchen wir sie?

Analogie: Stell dir eine Mappe mit mehreren Excel-Tabellen vor. Jede Tabelle sammelt eine Sorte Dinge (eine für Unternehmen, eine für Jobs, …). Jede Zeile ist ein Eintrag (ein bestimmter Job), jede Spalte ein Merkmal (Titel, Ort, Gehalt). Das Besondere gegenüber Excel: Die Tabellen sind über Nummern miteinander verknüpft („relational"), und ein Datenbanksystem wacht darüber, dass die Verknüpfungen stimmen – auch bei tausenden Zugriffen gleichzeitig.

Warum nicht einfach Variablen oder eine Textdatei?

  • Variablen im Server leben nur im Arbeitsspeicher – nach einem Neustart ist alles weg. Eine Datenbank schreibt auf die Festplatte: Bewerbungen bleiben erhalten.
  • Eine Datenbank kann gezielt suchen („alle aktiven Werkstudenten-Jobs in Berlin") ohne alles einzulesen, sie verhindert Widersprüche (z. B. eine Bewerbung auf einen Job, den es nicht gibt) und verkraftet gleichzeitige Zugriffe mehrerer Nutzer.

Warum MySQL? Es ist kostenlos, extrem verbreitet (viel Doku, viele Tools), zuverlässig – und von der Aufgabenstellung vorgesehen. Unsere Datenbank heißt studywork und ist auf utf8mb4 eingestellt, damit Umlaute (ä/ö/ü/ß) und Sonderzeichen korrekt gespeichert werden.

2. Wie spricht man mit einer Datenbank? SQL in 2 Minuten

Die Sprache heißt SQL. Vier Befehle decken fast alles ab – und sie spiegeln genau die REST-Methoden des Backends:

Lies SELECT * FROM jobs WHERE … einfach als Satz: „Wähle alle Spalten aus der Tabelle jobs, aber nur die Zeilen, bei denen …". Genau solche Sätze formuliert unser Backend – der Browser sieht davon nie etwas.

3. Primärschlüssel & Fremdschlüssel – der Kniff hinter „relational"

  • Primärschlüssel (PK): Jede Zeile bekommt eine eindeutige Nummer, bei uns immer die Spalte id, automatisch hochgezählt (1, 2, 3 …). Wie eine Matrikelnummer: unverwechselbar, auch wenn zwei Einträge sonst gleich aussehen.
  • Fremdschlüssel (FK): Eine Spalte, die die id einer anderen Tabelle enthält – der „Verweis-Pfeil" zwischen Tabellen.

Konkret: So hängen ein Unternehmen und seine Jobs zusammen:

companies                          jobs
┌────┬────────────────┐            ┌────┬────────────┬──────────────────────────────┐
│ id │ name           │            │ id │ company_id │ title                        │
├────┼────────────────┤            ├────┼────────────┼──────────────────────────────┤
│ 1  │ TechNova GmbH  │ ◄────┐     │ 1  │     1      │ Werkstudent Backend          │
│ 2  │ GreenLeaf …    │      └──── │ 2  │     1      │ Praktikum Data Science       │
└────┴────────────────┘            │ 4  │     2      │ Werkstudent Marketing        │
                                   └────┴────────────┴──────────────────────────────┘

Job 1 und Job 2 tragen company_id = 1 → beide gehören TechNova. Will die App „alle Jobs von TechNova", fragt sie: SELECT * FROM jobs WHERE company_id = 1. Die Datenbank erzwingt außerdem, dass dort nur Nummern stehen, die es in companies wirklich gibt – ein Job ohne existierendes Unternehmen ist unmöglich („referenzielle Integrität").

Design-Entscheidung: warum kein FK auf students?

Man kann sich bei StudyWork auch ohne Konto bewerben – das senkt die Hürde. Deshalb speichert eine Bewerbung Name und E-Mail als schlichten Text, statt fest auf einen Studenten-Eintrag zu zeigen. Hat der Bewerber doch ein Konto, erkennt das Backend das über die gleiche E-Mail-Adresse und kann z. B. dem Unternehmen den Profil-Link anzeigen oder dem Studenten „Meine Bewerbungen" auflisten. Eine bewusst „weiche" Verknüpfung – Flexibilität gegen etwas Strenge getauscht.

6. Das UML-/ER-Modell – vollständig in Worten

Ein UML-/ER-Diagramm ist die Landkarte einer Datenbank: Jede Tabelle ist ein Kasten (Entität) mit ihren Feldern, Beziehungen sind Linien zwischen den Kästen. An jedem Linienende steht eine Multiplizität – sie beantwortet: „Mit wie vielen Einträgen der anderen Seite kann dieser Eintrag verbunden sein?"

Lesehilfe: Multiplizitäten

  • 1 = genau eins (Pflicht)
  • 0..1 = keins oder eins
  • 0..* = beliebig viele, auch keins („*" = viele)
  • In der verbreiteten „Krähenfuß"-Notation zeichnet man das „viele"-Ende als kleine Gabel; das „viele"-Ende ist immer die Tabelle, die den Fremdschlüssel trägt.

Die Entitäten (Kästen mit allen Feldern)

  • COMPANY ( id, name, email, password_hash, description, website, created_at )
  • STUDENT ( id, name, email, password_hash, headline, bio, skills, study_program, university, location, website, profile_visibility, cv_filename, cv_data, cv_uploaded_at, created_at )
  • JOB ( id, company_id (FK), title, description, location, job_type, salary_range, status, created_at, updated_at )
  • APPLICATION ( id, job_id (FK), student_name, student_email, cover_letter, status, created_at )
  • JOB_REPORT ( id, job_id (FK), reason, message, created_at )
  • JOB_ALERT ( id, student_id (FK), title, location, job_type, created_at )

(Unterstrichen = Primärschlüssel; kursiv (FK) = Fremdschlüssel.)

Die Beziehungen – einzeln erklärt

BeziehungMultiplizitätSo liest man sie
COMPANY — JOB
„schreibt aus"		 COMPANY 1 : 0..* JOB Ein Unternehmen besitzt null bis viele Stellenanzeigen (auch ein Unternehmen ohne Anzeige ist gültig). Jede Anzeige gehört zu genau einem Unternehmen – nie zu keinem, nie zu zweien. Technisch: jobs.company_id → companies.id, Pflichtfeld (NOT NULL). Im Diagramm: Krähenfuß am JOB-Ende, Strich (1) am COMPANY-Ende.
JOB — APPLICATION
„erhält"		 JOB 1 : 0..* APPLICATION Ein Job kann null bis viele Bewerbungen erhalten; jede Bewerbung bezieht sich auf genau einen Job. Technisch: applications.job_id → jobs.id.
JOB — JOB_REPORT
„wird gemeldet"		 JOB 1 : 0..* JOB_REPORT Ein Job kann null bis viele Meldungen haben; jede Meldung betrifft genau einen Job. Technisch: job_reports.job_id → jobs.id.
STUDENT — JOB_ALERT
„speichert"		 STUDENT 1 : 0..* JOB_ALERT Ein Student kann null bis viele Such-Alerts speichern; jeder Alert gehört genau einem Studenten. Technisch: job_alerts.student_id → students.id.
STUDENT ⇢ APPLICATION
„hat eingereicht" (weich)		 STUDENT 0..1 : 0..* APPLICATION Die Besonderheit: Eine Bewerbung kann zu einem Studenten-Konto gehören, muss aber nicht (Gast-Bewerbung). Es gibt keinen Fremdschlüssel; die Zuordnung läuft über die Übereinstimmung applications.student_email = students.email. Im Diagramm zeichnet man so eine logische, nicht erzwungene Beziehung als gestrichelte Linie. Lesart: Ein Student hat null bis viele Bewerbungen; eine Bewerbung gehört zu keinem oder genau einem Studenten.

Damit ist das Modell auch sprachlich komplett beschreibbar – z. B.: „Ein Unternehmen hat beliebig viele Jobs; jeder Job gehört genau einem Unternehmen. Ein Job sammelt beliebig viele Bewerbungen und Meldungen. Ein Student besitzt beliebig viele Job-Alerts. Bewerbungen sind nur lose – über die E-Mail – mit Studentenkonten verknüpft, weil Bewerben auch ohne Konto möglich ist."

7. Löschen mit Kettenreaktion: ON DELETE CASCADE

Alle vier Fremdschlüssel tragen die Regel ON DELETE CASCADE: Wird der „Eltern"-Eintrag gelöscht, räumt die Datenbank die abhängigen „Kinder" automatisch mit weg.

Unternehmen TechNova löscht sein Konto
   └─→ alle Jobs von TechNova werden gelöscht          (FK jobs → companies)
         ├─→ alle Bewerbungen auf diese Jobs            (FK applications → jobs)
         └─→ alle Meldungen zu diesen Jobs              (FK job_reports → jobs)

Warum CASCADE?

Ohne die Regel blieben „Datenleichen" zurück: Bewerbungen auf Jobs, die es nicht mehr gibt. Solche verwaisten Einträge führen zu Fehlern und verletzen den Datenschutz-Gedanken („Konto gelöscht = zugehörige Daten weg"). Eine Ausnahme passt ins Bild: Löscht ein Student sein Konto, entfernt das Backend seine Bewerbungen selbst anhand der E-Mail – automatisch geht das nicht, weil hier ja bewusst kein Fremdschlüssel existiert.

8. Indizes – das Stichwortverzeichnis der Datenbank

Ein Index ist wie das Stichwortverzeichnis eines Buchs: Statt alle Seiten zu durchblättern, springt man direkt zur Fundstelle. Wir haben Indizes auf den Spalten angelegt, nach denen ständig gefiltert wird: jobs.company_id, jobs.job_type, jobs.status, applications.job_id, job_reports.job_id, job_alerts.student_id. Ergebnis: Die Jobsuche bleibt auch bei vielen Einträgen schnell.

9. Wie greift das Backend zu? (Anbindung)

Die komplette Verbindung steckt in backend/db.js – rund 40 Zeilen:

  • Connection Pool: Eine Datenbank-Verbindung aufzubauen kostet Zeit. Der „Pool" hält bis zu 10 Verbindungen offen und verleiht sie an Anfragen – wie ein Taxistand: Wagen warten bereit, statt für jede Fahrt ein neues Taxi zu bauen. Nach Gebrauch kommt die Verbindung zurück in den Pool.
  • Zugangsdaten aus .env: Host, Benutzer, Passwort und DB-Name stehen in einer lokalen Konfigurationsdatei, nie im Quellcode – so landen keine Geheimnisse im Git-Verlauf.
  • Start-Check: Beim Hochfahren „pingt" der Server die Datenbank an. Antwortet sie nicht, startet er gar nicht erst – mit einer klaren Meldung statt später rätselhafter Fehler.

Prepared Statements – der Schutz vor SQL-Injection

Das Angriffsszenario: Würde man Nutzereingaben direkt in SQL-Befehle hineinkleben, könnte ein Angreifer im Suchfeld statt eines Ortes so etwas eingeben: Berlin'; DELETE FROM jobs; -- – und aus der harmlosen Suche würde ein Befehl, der die Job-Tabelle leert. Das ist „SQL-Injection", einer der häufigsten Angriffe überhaupt.

// ❌ GEFÄHRLICH (machen wir NIE): Eingabe in den Befehl geklebt
"SELECT * FROM jobs WHERE location = '" + eingabe + "'"

// ✅ UNSER WEG: Platzhalter + Werte getrennt übergeben
pool.execute('SELECT * FROM jobs WHERE location = ?', [eingabe]);

Beim Prepared Statement bekommt die Datenbank zuerst den Befehl mit Platzhaltern (?) und dann die Werte separat. Sie behandelt die Eingabe dadurch strikt als Daten, nie als Befehl – der Angriffstext oben würde einfach wörtlich nach einem Ort namens „Berlin'; DELETE …" suchen (und nichts finden). Diese Technik verwenden wir bei ausnahmslos jeder Abfrage.

10. Einrichtung & Beispieldaten (db_setup.sql)

Das Skript backend/db_setup.sql baut die komplette Datenbank auf: Datenbank anlegen, alle 6 Tabellen erstellen, Indizes setzen, Beispieldaten einfüllen. Es ist idempotent – ein wichtiges Wort: Man kann es beliebig oft ausführen und erhält immer denselben frischen Ausgangszustand (es löscht die Tabellen vorher und legt sie neu an). Perfekt zum Zurücksetzen nach Experimenten – aber Achtung: eigene Test-Daten sind danach weg.

Danach enthält die Datenbank: 6 Unternehmen, 1 Studenten-Demokonto (Lena Hofmann, mit ausgefülltem Profil), 20 Stellenanzeigen (17 aktiv, mit gestreuten Datumsangaben – die Seite wirkt dadurch „lebendig"), 5 Bewerbungen und 1 Job-Alert. Alle Demo-Logins nutzen das Passwort studywork123.

13. Typische Fragen – kurze Antworten

Warum überhaupt eine Datenbank – der Server könnte sich Daten doch merken?
Nur bis zum nächsten Neustart. Eine Datenbank schreibt auf die Festplatte, kann gezielt suchen, verhindert Widersprüche und verkraftet parallele Zugriffe.
Warum hat applications keinen Fremdschlüssel auf students?
Weil Bewerben ohne Konto möglich sein soll. Die Zuordnung zu einem Konto geschieht „weich" über die identische E-Mail-Adresse.
Was passiert, wenn ein Unternehmen gelöscht wird?
CASCADE-Kettenreaktion: seine Jobs → deren Bewerbungen und Meldungen werden automatisch mitgelöscht.
Kann in status irgendein Text stehen?
Nein – es ist ein ENUM: Die Datenbank selbst akzeptiert nur die vorgegebenen Werte (z. B. aktiv/pausiert/geschlossen).
Ist es nicht unsicher, den Lebenslauf in der Datenbank zu speichern?
Die PDF liegt als Text kodiert in einer Spalte und wird nur über eine zugriffsgeschützte Backend-Route ausgeliefert (nur berechtigte Unternehmen). Für ein Produkt mit großen Dateien würde man einen Datei-Speicher nutzen; für unsere Demo ist die DB-Lösung einfach und völlig ausreichend.
Port 3000 oder 3306 – was ist was?
3000 = unser Web-Server (Backend + Seiten), 3306 = der MySQL-Dienst. Zwei getrennte Programme, zwei „Zimmernummern".
Woher „weiß" die Jobsuche so schnell die Treffer?
Durch Indizes auf den Filter-Spalten – die DB springt direkt zu den passenden Zeilen, statt alles zu durchsuchen.

🗣️ So erklärst du die Datenbank in 5 Sätzen weiter

  1. Alle Daten von StudyWork liegen dauerhaft in einer MySQL-Datenbank namens studywork – organisiert in 6 verknüpften Tabellen (Unternehmen, Studenten, Jobs, Bewerbungen, Meldungen, Job-Alerts).
  2. Verknüpft wird über Schlüssel: Jede Zeile hat eine eindeutige id (Primärschlüssel), und Fremdschlüssel wie jobs.company_id zeigen darauf – so gehört jeder Job zu genau einem Unternehmen, das selbst beliebig viele Jobs haben kann (1 : 0..*).
  3. Einzige Besonderheit: Bewerbungen zeigen nicht fest auf Studenten, sondern werden nur über die gleiche E-Mail zugeordnet – damit man sich auch ohne Konto bewerben kann.
  4. Beim Löschen räumt ON DELETE CASCADE abhängige Daten automatisch mit weg, Indizes halten die Suche schnell, und das Setup-Skript baut die ganze DB samt Beispieldaten jederzeit neu auf.
  5. Zugriff hat ausschließlich das Backend – über einen Verbindungs-Pool und ausnahmslos mit Prepared Statements, die SQL-Injection unmöglich machen.
