Datenbank – das „Lager" aller Daten

Wie wir die Daten dauerhaft speichern, wie die Tabellen zusammenhängen (UML in Worten) und wie das Backend darauf zugreift.

In einem Satz

Wir nutzen MySQL – ein Datenbanksystem, das Daten in Tabellen speichert (wie viele verbundene Excel-Tabellen). Unsere Datenbank heißt studywork und besteht aus 6 Tabellen. Nur das Backend spricht mit ihr.

Was ist eine (relationale) Datenbank?

Stell dir eine Sammlung von Excel-Tabellen vor. Jede Tabelle speichert eine Sorte Dinge (eine Tabelle für Unternehmen, eine für Jobs, …). Jede Zeile ist ein Eintrag (z. B. ein bestimmter Job), jede Spalte ein Merkmal (z. B. der Titel). Das Besondere: Die Tabellen sind miteinander verbunden – ein Job „weiß", zu welchem Unternehmen er gehört.

  • Warum überhaupt eine Datenbank? Weil Daten dauerhaft bleiben müssen – auch wenn man den Server neu startet oder den Computer ausschaltet.
  • Warum MySQL? Es ist kostenlos, sehr verbreitet, zuverlässig und von der Aufgabenstellung als Datenbank gewünscht.
  • Zeichensatz utf8mb4: damit Umlaute (ä, ö, ü, ß) korrekt gespeichert werden.

Zwei wichtige Begriffe: Primärschlüssel & Fremdschlüssel

  • Primärschlüssel (PK): eine eindeutige Nummer, die jede Zeile unverwechselbar macht. Bei uns heißt sie immer id und wird automatisch hochgezählt (1, 2, 3, …).
  • Fremdschlüssel (FK): eine Spalte, die auf die id einer anderen Tabelle zeigt und so die Verbindung herstellt. Beispiel: jobs.company_id zeigt auf companies.id – so „gehört" jeder Job zu einem Unternehmen.

Das UML-/ER-Modell in Worten

Ein UML-/ER-Diagramm zeigt die Tabellen als Kästen (Entitäten) mit ihren Feldern und verbindet sie mit Linien (Beziehungen). An den Linienenden stehen Multiplizitäten – sie sagen, „wie viele" auf jeder Seite zusammengehören.

Lesehilfe für Multiplizitäten

  • 1 = genau eins
  • 0..* = beliebig viele (auch keine) – „*" heißt „viele"
  • 0..1 = keins oder eins

Die Entitäten (Kästen mit Feldern)

  • COMPANY ( id, name, email, password_hash, description, website, created_at )
  • STUDENT ( id, name, email, password_hash, headline, bio, skills, study_program, university, location, website, profile_visibility, cv_filename, cv_data, cv_uploaded_at, created_at )
  • JOB ( id, company_id (FK), title, description, location, job_type, salary_range, status, created_at, updated_at )
  • APPLICATION ( id, job_id (FK), student_name, student_email, cover_letter, status, created_at )
  • JOB_REPORT ( id, job_id (FK), reason, message, created_at )
  • JOB_ALERT ( id, student_id (FK), title, location, job_type, created_at )

(Unterstrichen = Primärschlüssel, kursiv (FK) = Fremdschlüssel.)

Die Beziehungen (Linien mit Richtung & Multiplizität)

„hat / schreibt aus"		 1 : 0..* Ein Unternehmen hat 0 bis viele Stellenanzeigen; jede Anzeige gehört zu genau einem Unternehmen. Umgesetzt über den Fremdschlüssel jobs.company_id → companies.id. In Krähenfuß-Notation: „viele"-Symbol am JOB-Ende, einfacher Strich am COMPANY-Ende. Pflichtbeziehung (jeder Job muss ein Unternehmen haben).
JOB → APPLICATION
„erhält"		 1 : 0..* Ein Job hat 0 bis viele Bewerbungen; jede Bewerbung gehört zu genau einem Job. Fremdschlüssel applications.job_id → jobs.id.
JOB → JOB_REPORT
„wird gemeldet als"		 1 : 0..* Ein Job kann 0 bis viele Meldungen haben; jede Meldung betrifft genau einen Job. Fremdschlüssel job_reports.job_id → jobs.id.
STUDENT → JOB_ALERT
„speichert"		 1 : 0..* Ein Student hat 0 bis viele gespeicherte Such-Alerts; jeder Alert gehört zu genau einem Studenten. Fremdschlüssel job_alerts.student_id → students.id.
STUDENT ⇠ APPLICATION
(weiche Beziehung, ohne Fremdschlüssel)		 0..1 : 0..* Eine Bewerbung kann zu einem Studenten-Konto gehören – muss aber nicht (Bewerbung ohne Konto ist erlaubt). Deshalb gibt es hier keinen Fremdschlüssel; verbunden wird über die gleiche E-Mail-Adresse (applications.student_email = students.email). Im Diagramm zeichnet man das als gestrichelte Linie. Multiplizität: Ein Student hat 0..* Bewerbungen (alle mit seiner E-Mail); eine Bewerbung gehört zu 0 oder 1 Student.

Kurz als „Text-Diagramm"

COMPANY  1 ───< 0..*  JOB  1 ───< 0..*  APPLICATION
                      JOB  1 ───< 0..*  JOB_REPORT
STUDENT  1 ───< 0..*  JOB_ALERT
STUDENT  0..1 ┄┄< 0..* APPLICATION   (weich, über die E-Mail-Adresse)

Das Zeichen < ist der „Krähenfuß" und steht für das „viele"-Ende.

Was passiert beim Löschen? (ON DELETE CASCADE)

Alle Fremdschlüssel haben die Regel „CASCADE": Wird ein „Eltern"-Eintrag gelöscht, verschwinden automatisch die abhängigen „Kinder".

  • Löscht ein Unternehmen sein Konto → alle seine Jobs werden gelöscht → dadurch auch deren Bewerbungen und Meldungen (Kettenreaktion).
  • Löscht ein Student sein Konto → seine Job-Alerts werden gelöscht (seine Bewerbungen entfernt das Backend zusätzlich anhand der E-Mail).

Warum CASCADE?

Damit keine „verwaisten" Daten übrig bleiben – z. B. eine Bewerbung zu einem Job, den es gar nicht mehr gibt. Die Datenbank hält sich so immer selbst sauber.

Wie das Backend an die Daten kommt (Anbindung)

  • Die Verbindung steht in backend/db.js. Wir nutzen ein „Verbindungs-Bündel" (Connection Pool): Statt für jede Anfrage neu zur Datenbank zu verbinden (langsam), werden mehrere Verbindungen bereitgehalten und wiederverwendet.
  • Die Zugangsdaten (Adresse, Benutzer, Passwort, DB-Name) stehen in einer .env-Datei – nie im Code.
  • Alle Abfragen laufen als Prepared Statements: Werte werden sicher als Parameter übergeben, nicht in den Befehl „geklebt". Das schützt vor SQL-Injection (dem Versuch, die Datenbank über Eingabefelder auszutricksen).
// vereinfachtes Beispiel (im Backend)
const [zeilen] = await pool.execute(
  'SELECT * FROM jobs WHERE job_type = ? AND status = ?',
  ['Werkstudent', 'aktiv']   // Werte sicher als Parameter
);

Wie schaue ich selbst in die Datenbank? (GUI)

Zwei Wege – Kommandozeile oder ein grafisches Programm:

a) Kommandozeile

mysql -u root studywork        # verbinden
SHOW TABLES;                    # alle Tabellen
SELECT * FROM companies;        # alle Unternehmen ansehen
DESCRIBE jobs;                  # Spalten der Tabelle jobs

b) Grafisches Programm (empfehlenswert)

z. B. MySQL Workbench, DBeaver, TablePlus oder phpMyAdmin. Damit klickt man sich durch die Tabellen wie durch Ordner.

Verbindungsdaten für die GUI

  • Host: localhost · Port: 3306
  • Benutzer / Passwort: wie in der .env (lokal z. B. root ohne Passwort)
  • Datenbank/Schema: studywork

Einrichtung & Beispieldaten

Das Skript backend/db_setup.sql legt die Datenbank, alle Tabellen und Beispieldaten an. Es ist idempotent (löscht vorhandene Tabellen zuerst und baut sie neu) – man kann es also gefahrlos erneut ausführen, um den Ausgangszustand wiederherzustellen.

mysql --default-character-set=utf8mb4 -u root < backend/db_setup.sql

Danach enthält die Datenbank Demo-Inhalte: 6 Unternehmen, ein Studenten-Konto, 20 Stellenanzeigen und einige Bewerbungen – damit die Seite gleich „lebendig" aussieht.
