Frontend – was der Nutzer sieht und bedient

Wie die Oberfläche von StudyWork gebaut ist, wie sie an ihre Daten kommt – und warum wir jede Entscheidung so getroffen haben.

🎯 Nach dieser Seite kannst du erklären:

  • was HTML, CSS und JavaScript jeweils tun (mit Mini-Beispielen),
  • wie unsere Seiten aufgebaut sind und wie sie Daten vom Server holen (fetch + JSON),
  • was die zentrale Datei api.js macht und warum es sie gibt,
  • wie Formular-Prüfung, Login-Anzeige, Dark-Mode und Responsive Design funktionieren,
  • was XSS ist und wie wir uns davor schützen.

1. Die drei Bausteine: HTML, CSS, JavaScript

Haus-Analogie: HTML ist der Rohbau (Wände, Räume, Türen), CSS ist Einrichtung und Anstrich (Farben, Möbel, Stil), JavaScript ist die Elektrik (Lichtschalter, Türöffner – alles, was auf Aktionen reagiert).

HTML – die Struktur (was steht auf der Seite?)

HTML besteht aus „Tags" – Markierungen in spitzen Klammern, die Inhalte auszeichnen:

<h1>Jobs finden</h1>
<input type="search" id="f-title" placeholder="Stichwort, z. B. Marketing" />
<button type="submit">Suchen</button>

<h1>…</h1> – eine Überschrift erster Ordnung.

<input type="search"> – ein Eingabefeld; das id ist sein „Name", über den JavaScript es später findet.

<button> – ein Knopf. Mehr ist HTML im Kern nicht: Inhalte benennen.

CSS – das Aussehen (wie sieht es aus?)

CSS besteht aus Regeln: „Wähle Elemente aus → gib ihnen Eigenschaften."

.btn-primary {
  background: #0e7c52;   /* grüne Füllung */
  color: white;          /* weiße Schrift */
  border-radius: 999px;  /* runde Ecken */
}

.btn-primary – „alle Elemente mit der Klasse btn-primary" (unsere grünen Hauptknöpfe).

Die Zeilen darunter sind Eigenschaften: Hintergrundfarbe, Schriftfarbe, Eckenrundung.

Unser gesamtes Design steckt in einer Datei: frontend/css/style.css.

JavaScript – das Verhalten (was passiert bei Aktionen?)

button.addEventListener('click', () => {
  ladeJobs();   // wenn geklickt wird: Jobs laden
});

addEventListener('click', …) heißt wörtlich: „Höre auf das Ereignis Klick und führe dann diesen Code aus." So reagiert die Seite auf den Nutzer.

JavaScript kann außerdem die Seite umbauen (Inhalte einfügen/entfernen) und mit dem Server reden – dazu gleich mehr.

Warum „pures" JavaScript und kein Framework (React, Vue …)?

Erstens verlangt die Aufgabenstellung genau diesen Grundstack. Zweitens ist unser Projekt überschaubar genug, dass ein Framework mehr verdecken als helfen würde: Ohne Framework sieht man in jedem Schritt, was wirklich passiert – ideal zum Lernen, Erklären und Bewerten. Der Preis: Wir mussten ein paar Dinge selbst bauen (z. B. die gemeinsame Navigation), was Frameworks sonst abnehmen. Für diese Projektgröße ist das ein guter Tausch.

2. Aufbau: eine klassische Mehrseiten-Website

StudyWork ist eine Multi-Page-App (MPA): Jeder Bereich ist eine eigene HTML-Datei. Klickt man einen Link, lädt der Browser die nächste Seite. (Die Alternative wäre eine „Single-Page-App", bei der eine einzige Seite alles per JavaScript nachlädt – mächtiger, aber deutlich komplexer. Für unsere klar getrennten Bereiche ist die MPA die einfachere, ehrlichere Wahl.)

frontend/ ├── index.html ← Startseite (Suchfeld, neueste Jobs, Statistiken) ├── jobs.html ← Jobsuche: Liste + Filter + Sortierung ├── job-detail.html ← eine Stelle im Detail + Bewerbungsformular ├── company-dashboard.html ← Bereich für Unternehmen (nur mit Login) ├── profile.html ← Studenten-Profil, Lebenslauf, Bewerbungen, Job-Alerts ├── login.html / register.html ← Anmelden / Konto erstellen ├── 404.html ← freundliche Fehlerseite ├── css/style.css ← das komplette Aussehen (inkl. Dark-Mode) └── js/ ├── api.js ← Grundbaustein jeder Seite (Details unten) ├── jobs.js ← Logik für Jobsuche + Jobdetail ├── company.js ← Logik für das Unternehmens-Dashboard ├── auth.js ← Logik für Login & Registrierung └── profile.js ← Logik für das Studenten-Profil

Jede HTML-Seite lädt am Ende zwei Skripte – immer zuerst api.js, dann ihr eigenes Modul:

<script src="js/api.js" defer></script>
<script src="js/jobs.js" defer></script>

(defer heißt: „führe das Skript erst aus, wenn die Seite fertig gelesen ist" – sonst würde JavaScript nach Elementen suchen, die es noch gar nicht gibt.) Woran erkennt jobs.js, ob es gerade die Suche oder die Detailseite bedient? Am <body data-page="jobs">-Attribut – eine kleine Markierung im HTML.

3. Wie kommt die Seite an ihre Daten? (fetch + JSON)

Eine HTML-Datei ist zunächst „leer" – die Jobliste steht nicht im HTML, sondern kommt aus der Datenbank. Den Transport übernimmt fetch, eine eingebaute Browser-Funktion, die im Hintergrund eine HTTP-Anfrage an den Server schickt (ohne die Seite neu zu laden):

const antwort = await fetch('/api/v1/jobs?location=Berlin');
const paket   = await antwort.json();
// paket = { success: true, data: [ {id: 1, title: "Werkstudent Backend", …}, … ] }

Zeile 1: Anfrage an das Backend: „Gib mir alle Jobs in Berlin." await heißt „warte, bis die Antwort da ist".

Zeile 2: Die Antwort ist Text im JSON-Format – .json() macht daraus ein JavaScript-Objekt, mit dem der Code arbeiten kann.

Zeile 3: So sieht das Paket aus: ein Erfolgs-Kennzeichen und die eigentlichen Daten (eine Liste von Jobs).

Das Herzstück: api.js

Damit nicht jede Seite diese Technik selbst nachbauen muss, gibt es einen zentralen Baustein, der auf jeder Seite geladen wird. Er übernimmt vier Dinge:

  1. Der API-Client: sprechende Funktionen für jede Server-Anlaufstelle. Die Seiten schreiben nicht selbst fetch(…), sondern z. B.: 
    StudyWork.API.jobs.list({ location: 'Berlin' })  // Jobs suchen
StudyWork.API.applications.create({ … })         // Bewerbung absenden
StudyWork.API.auth.login({ … })                  // anmelden
    Intern steckt dahinter genau ein fetch-Wrapper, der auch die Fehlerbehandlung vereinheitlicht: Meldet der Server success: false, wird daraus automatisch eine verständliche Fehlermeldung für den Nutzer.
  2. Navigation & Fußzeile: werden per JavaScript auf jeder Seite eingefügt – so existiert das Menü nur einmal im Code statt achtmal kopiert (weniger Fehler, eine Änderung wirkt überall).
  3. Login-Anzeige: Beim Laden fragt jede Seite den Server „Wer bin ich?" (GET /api/v1/auth/me). Je nach Antwort zeigt die Navigation „Anmelden/Registrieren" oder den Namen mit „Abmelden".
  4. UI-Helfer: Erfolgs-/Fehlermeldungen (kleine Einblendungen, „Toasts"), Datumsformatierung („vor 3 Tagen"), der Hell/Dunkel-Umschalter, die Job-Kachel als wiederverwendbare Vorlage u. v. m.

Warum eine zentrale api.js?

Das Prinzip heißt „Don't repeat yourself" (wiederhole dich nicht). Alles, was mehrere Seiten brauchen, existiert genau einmal. Vorteil: Ändert sich z. B. das Antwortformat des Servers, passen wir eine Datei an – nicht fünf. Und die Seitenlogik bleibt lesbar, weil sie nur noch sagt was sie will, nicht wie der Transport funktioniert.

4. Walkthrough 1: Die Jobsuche (jobs.html + jobs.js)

  1. Platzhalter zeigen: Direkt beim Laden erscheinen graue „Skeleton"-Kacheln. So wirkt die Seite nie leer oder eingefroren, während Daten unterwegs sind.
  2. Daten holen: API.jobs.list(filter) fragt das Backend; dabei werden die aktuellen Filter (Stichwort, Ort, Anstellungsart) als Parameter mitgeschickt. Studenten sehen dabei grundsätzlich nur aktive Stellen.
  3. Anzeigen: Für jeden Job baut das JavaScript eine Kachel (Titel, Firma, Ort, Gehalt, „Neu"-Abzeichen bei Stellen < 7 Tage) und fügt sie in die Seite ein.
  4. Live-Suche: Tippt man ins Suchfeld, wird automatisch neu gesucht – mit einer kleinen Verzögerung („Debounce"): Erst wenn man ~0,3 s nicht mehr tippt, geht die Anfrage raus. Sonst würde bei „Marketing" für M, Ma, Mar … je eine eigene Anfrage entstehen.
  5. Teilbare Links: Die aktiven Filter werden in die Adresszeile geschrieben (z. B. jobs.html?title=Marketing). Diesen Link kann man kopieren und verschicken – der Empfänger sieht exakt dieselbe gefilterte Suche.
  6. Extras: „Nur Remote"-Schalter, Sortierung, Merkliste per Herz (lokal im Browser gespeichert), leere Zustände mit freundlichem Hinweis statt weißer Fläche.

5. Walkthrough 2: Das Bewerbungsformular (unser Lena-Beispiel)

  1. Auf der Detailseite steht das Formular: Name, E-Mail, Anschreiben (mit Zeichenzähler). Ist Lena eingeloggt, sind Name und E-Mail schon aus ihrem Konto vorausgefüllt. Hat sie sich auf diese Stelle schon beworben, erscheint statt des Formulars ein Hinweis mit dem aktuellen Status ihrer Bewerbung.
  2. Client-Prüfung beim Absenden: Pflichtfelder leer? E-Mail ohne @? Dann zeigt das Frontend die Fehler direkt am jeweiligen Feld an – schnell und ohne Server-Kontakt.
  3. Ist alles plausibel, schickt API.applications.create(…) die Daten als POST-Anfrage ans Backend. Der Absenden-Knopf wird währenddessen deaktiviert und zeigt „Wird gesendet…" – damit niemand doppelt klickt.
  4. Kommt Erfolg zurück, wird das Formular durch eine Bestätigung ersetzt. Kommt ein Fehler (z. B. „bereits beworben" – das weiß nur der Server), erscheint die Meldung des Servers als Einblendung, und Lena kann korrigieren.

Warum prüfen Frontend UND Backend?

Die Frontend-Prüfung ist Komfort (sofortiges Feedback, keine unnötigen Anfragen). Verbindlich ist sie nicht: Jeder kann im Browser die Prüfungen aushebeln oder Anfragen direkt an den Server schicken. Deshalb prüft das Backend alles noch einmal – das ist die echte Kontrolle. Merksatz: Frontend-Prüfung ist Höflichkeit, Backend-Prüfung ist Sicherheit.

6. Login im Frontend: woher weiß die Seite, wer ich bin?

Nach dem Anmelden bekommt der Browser vom Server ein Sitzungs-Cookie (eine Art Eintrittsstempel – Details auf der Backend-Seite). Wichtig fürs Frontend:

  • Der Browser schickt dieses Cookie automatisch bei jeder Anfrage mit – unser JavaScript muss (und kann!) es nicht anfassen: Es ist als „HttpOnly" markiert, also für JavaScript unsichtbar. Das schützt es vor Diebstahl durch eingeschleusten Code.
  • Die Seite erfährt nur über die Frage GET /auth/me, wer eingeloggt ist, und passt die Oberfläche an (Name in der Navigation, Dashboard-Zugang, vorausgefüllte Formulare).
  • Abmelden führt immer zurück zur Startseite, damit keine geschützten Inhalte (z. B. ein Profil) sichtbar stehen bleiben, und zeigt eine Bestätigung an.

7. Design: responsive, Dark-Mode, Barrierefreiheit

  • Corporate Design: Ein fester Farb- und Schriftbaukasten („Ink & Amber": Creme-Hintergrund, dunkle Konturen, Grün/Bernstein als Akzente) ist als CSS-Variablen ganz oben in style.css definiert – Farben ändert man an genau einer Stelle, und die ganze App zieht mit.
  • Responsive: Mit „Media Queries" (CSS-Regeln, die nur ab/unter bestimmten Bildschirmbreiten gelten) ordnet sich das Layout um: Auf dem Handy stapeln sich die Job-Kacheln untereinander, die Navigation bricht sauber um; auf dem Desktop gibt es mehrere Spalten. Getestet für ~375 px (Handy), 768 px (Tablet) und 1280 px+ (PC).
  • Dark-Mode: Der Umschalter setzt ein Attribut am Seitenanfang (data-theme="dark"); das CSS tauscht daraufhin nur die Farbvariablen aus. Die Wahl wird im localStorage gespeichert – einem kleinen Merkzettel des Browsers, der auch nach dem Schließen erhalten bleibt (dort liegt auch die Merkliste).
  • Barrierefreiheit: aussagekräftige Beschriftungen für Vorlese-Software (ARIA-Labels an Icon-Knöpfen), Bedienbarkeit per Tastatur, und in Dialogen bleibt der Fokus „gefangen", bis man sie schließt.

8. Sicherheit im Frontend: Was ist XSS?

XSS („Cross-Site-Scripting") ist ein Angriff, bei dem jemand Code als Inhalt einschleust. Beispiel: Ein Angreifer legt einen Job an mit dem Titel <script>stehleDaten()</script>. Würden wir diesen Titel ungefiltert in die Seite einbauen, würde der Browser ihn als Programm ausführen – bei jedem Besucher!

Unser Schutz: Jeder Text, der von außen kommt (aus der Datenbank, von Nutzern), läuft vor der Anzeige durch eine Funktion escapeHtml(). Sie ersetzt gefährliche Zeichen durch harmlose Darstellungen (aus < wird &lt;). Ergebnis: Der Browser zeigt den Text nur an, statt ihn auszuführen. Diese Regel gilt bei uns ausnahmslos – kein Nutzertext landet ungefiltert im HTML.

10. Typische Fragen – kurze Antworten

Warum kein React/Vue?
Aufgabenstellung + Projektgröße: Purer Code ist hier durchschaubarer; ein Framework würde mehr verdecken als nützen.
Woher weiß eine Seite, welche Logik sie ausführen soll?
Über <body data-page="…"> – das JavaScript liest diese Markierung und startet den passenden Teil.
Warum sieht man beim Laden kurz graue Kästen?
Das sind Skeleton-Platzhalter: Sie zeigen „hier kommt gleich etwas" statt einer leeren Fläche, während die Daten vom Server unterwegs sind.
Kann das Frontend Nutzer wirklich „schützen"?
Nein – es kann nur helfen und führen. Alles Sicherheitsrelevante entscheidet das Backend; das Frontend könnte von jedem Nutzer manipuliert werden.
Was passiert, wenn der Server nicht erreichbar ist?
Der fetch-Wrapper fängt das ab und zeigt eine verständliche Meldung („Netzwerkfehler – der Server ist nicht erreichbar") statt einer kaputten Seite.

🗣️ So erklärst du das Frontend in 5 Sätzen weiter

  1. Das Frontend sind die Web-Seiten im Browser, gebaut aus HTML (Struktur), CSS (Aussehen) und JavaScript (Verhalten) – bewusst ohne Framework, damit alles nachvollziehbar bleibt.
  2. Jeder Bereich ist eine eigene HTML-Seite; eine zentrale Datei api.js lädt auf allen Seiten und übernimmt den Datenaustausch mit dem Server, die Navigation und die Login-Anzeige.
  3. Daten holt die Seite per fetch vom Backend – als JSON – und baut daraus z. B. die Job-Kacheln; beim Tippen wird live (mit Debounce) neu gesucht, Filter landen teilbar in der URL.
  4. Formulare werden doppelt geprüft: im Frontend für schnelles Feedback, verbindlich aber noch einmal im Backend.
  5. Design: einheitliche Farbvariablen, responsive für Handy bis Desktop, Dark-Mode per Umschalter – und alle Fremdtexte werden vor der Anzeige „escaped", um XSS-Angriffe zu verhindern.
