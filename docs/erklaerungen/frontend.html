Frontend – was der Nutzer sieht

Wie wir die sichtbare Web-Oberfläche gebaut haben und warum.

In einem Satz

Das Frontend sind die Web-Seiten im Browser. Wir haben sie mit den drei Grund-Bausteinen des Webs gebaut: HTML (Struktur), CSS (Aussehen) und JavaScript (Verhalten). Das JavaScript holt sich die Daten vom Backend und zeigt sie an.

Die drei Bausteine – mit einer Analogie

Denk an ein Haus: HTML ist der Rohbau (Wände, Räume, Türen), CSS ist die Einrichtung und der Anstrich (Farben, Möbel, Stil) und JavaScript ist die Elektrik (Lichtschalter, Bewegungsmelder – alles, was auf Aktionen reagiert).

  • HTML legt fest, was auf der Seite steht: Überschriften, Textfelder, Knöpfe, Listen. Jede Seite ist eine eigene .html-Datei.
  • CSS legt fest, wie es aussieht: Farben, Abstände, Schriften, und dass es auf Handy, Tablet und PC gut aussieht. Alles in einer Datei: css/style.css.
  • JavaScript legt fest, was passiert: Klick auf „Suchen", Formular absenden, Daten laden und anzeigen. Aufgeteilt in mehrere Dateien unter js/.

Warum „pures" JavaScript und kein Framework (wie React/Vue)?

Weil die Aufgabenstellung genau das verlangt (HTML/CSS/JavaScript) und unser Projekt überschaubar ist. Ohne Framework bleibt der Code durchschaubar – man sieht genau, was passiert, ohne eine zusätzliche „Magie"-Ebene. Das ist auch fürs Verstehen und Präsentieren einfacher.

Wie die Seiten aufgebaut sind

Es ist eine klassische Mehrseiten-Website: Für jeden Bereich gibt es eine eigene HTML-Datei. Klickt man einen Link, lädt der Browser die nächste Seite.

frontend/ ├── index.html ← Startseite ├── jobs.html ← Jobsuche (Liste + Filter) ├── job-detail.html ← eine Stelle im Detail + Bewerbungsformular ├── company-dashboard.html ← Bereich für Unternehmen ├── profile.html ← Studenten-Profil ├── login.html / register.html ← Anmelden / Registrieren ├── 404.html ← Fehlerseite (Seite nicht gefunden) ├── css/style.css ← das komplette Aussehen └── js/ ← das Verhalten (JavaScript) ├── api.js ← Grundbaustein: redet mit dem Backend, lädt überall ├── jobs.js ← Logik für Jobsuche + Jobdetail ├── company.js ← Logik für das Unternehmens-Dashboard ├── auth.js ← Logik für Login & Registrierung └── profile.js ← Logik für das Studenten-Profil

Das Herzstück: api.js

Diese Datei wird auf jeder Seite geladen. Sie hat vier Aufgaben:

  1. Mit dem Backend reden. Immer wenn die Seite Daten braucht (z. B. „alle Jobs"), fragt sie über api.js beim Backend an. Der Fachbegriff dafür ist fetch – eine eingebaute Browser-Funktion, die eine Anfrage übers Netz schickt.
  2. Navigation & Fußzeile einbauen. Damit oben überall dieselbe Menüleiste erscheint (ohne sie auf jeder Seite zu kopieren).
  3. Anzeigen, wer eingeloggt ist. Beim Laden fragt die Seite „Wer bin ich?" beim Backend und zeigt dann entweder „Anmelden" oder den Namen + „Abmelden".
  4. Hell-/Dunkel-Modus und kleine Helfer (z. B. Erfolgs-/Fehlermeldungen).

Warum eine zentrale api.js?

Damit die einzelnen Seiten nicht jedes Mal die technischen Details des Datenaustauschs kennen müssen. Sie rufen einfach z. B. API.jobs.list() auf – der Rest passiert im Hintergrund. Das vermeidet doppelten Code und Fehler.

Wie eine Seite ihre Daten bekommt (Beispiel Jobsuche)

  1. Die Seite lädt und zeigt zunächst graue Platzhalter (damit es nicht leer wirkt).
  2. Das JavaScript fragt beim Backend nach den Jobs.
  3. Das Backend holt sie aus der Datenbank und schickt sie als Daten zurück.
  4. Das JavaScript baut aus jedem Job eine „Kachel" und setzt sie in die Seite.
  5. Tippt man ins Suchfeld, wird automatisch neu gesucht; die Filter landen in der Adresszeile (URL), sodass man einen Suchlink teilen kann.

Worauf wir beim Aussehen (Design) geachtet haben

  • Einheitlicher Look („Corporate Design"): feste Farben (Creme, Grün, Bernstein), ein Logo, klare Schriften – überall gleich.
  • Responsive: funktioniert auf Handy, Tablet und Desktop; das Menü bricht auf kleinen Bildschirmen sauber um.
  • Hell-/Dunkel-Modus per Umschalter.
  • Gute Rückmeldungen: Ladeanzeigen, freundliche „nichts gefunden"-Hinweise, Erfolgs-/Fehlermeldungen als kleine Einblendungen.

Sicherheit im Frontend (einfach erklärt)

  • Schutz vor Schadcode (XSS): Texte, die aus der Datenbank kommen (z. B. ein Jobtitel), werden vor der Anzeige „entschärft". So kann niemand über einen manipulierten Text heimlich Programmcode einschleusen.
  • Die eigentliche Rechteprüfung passiert im Backend. Das Frontend blendet Dinge nur der Bequemlichkeit halber aus – verlassen kann man sich nur auf den Server. (Warum? Weil man im Browser alles manipulieren könnte – die verbindliche Kontrolle muss deshalb hinter den Kulissen sitzen.)
